# 第一轮：字面证据提取

## 目标

只提取题目明确表达的内容，建立完整要求台账。不要解释题意、补充常识或推荐方法。

## 必须执行

1. 按段落、问题编号、表格、图和附件说明逐项扫描。
2. 对每个具有信息价值的陈述建立 `req-*` 条目。
3. 标记为 `background`、`fact`、`input`、`objective`、`constraint`、`output` 或 `evaluation`。
4. 记录最具体的 `source_ref`，保留限定词和量词。
5. 对 `must`、`should`、`analyze`、`compare`、`evaluate`、`justify`、`建议`、`分析`、`讨论` 等动作词逐项检查是否形成交付要求。

## 提示问题

- 题面明确给出了哪些对象、数值、单位、时间和空间范围？
- 用户最终必须回答、比较、预测、评价、解释或提交什么？
- 哪些条件使用了“仅、至少、至多、原则上、假定、在……情况下”等限定？
- 图、表、脚注和附录是否包含正文未重复的要求？
- 哪些语句只是背景，不能当作约束或事实？

## 输出

更新 `requirement-trace.yaml`。此轮不填写具体模型名称，也不把推断写入事实条目。

## 失败信号

- 大段题面只对应一个要求条目。
- `source_ref` 只有文件名，没有页码、段落或工作表锚点。
- 原题要求的解释、比较或建议任务没有被登记。
- 使用“显然”“通常”“应该是”补充题面未给出的事实。
