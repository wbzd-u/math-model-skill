# 第三轮：要求与子问题拆解

## 目标

将自然语言要求重写为边界清楚、可验证的子问题，不提前决定解法。

## 拆解公式

每个子问题都按以下结构表达：

```text
动作 + 对象 + 条件 + 评价标准 + 必须输出
```

## 必须执行

1. 将每个 `objective`、`output` 和 `evaluation` 要求映射到至少一个子问题。
2. 合并重复表达，但保留全部来源要求 ID。
3. 将一句话中的多个独立动作拆开。
4. 识别没有题号但需要执行的隐藏任务，例如解释、比较、检验和建议。
5. 为每个子问题填写输入、输出、目标、约束、评价标准和未解决项。
6. 为每个子问题形成初步 `task_tags` 和 `semantic_contract`；随后按 manifest 加载匹配结构检查卡并补全其内容。

## 提示问题

- 这个任务的动作动词是什么？
- 操作对象和适用条件是什么？
- 怎样的结果才算回答了题目？
- 需要给出数值、排序、方案、解释、图表还是政策建议？
- 是否包含“先分析再决策”“先估计再预测”等两个任务？
- 两个题问是独立、递进、扩展情景还是同一任务的不同评价？
- 任何有效答案必须满足的成功谓词和验收条件分别是什么？

## 输出

填写 `subproblems.yaml`，并完成 `requirement-trace.yaml` 中的子问题映射。

## 失败信号

- 子问题名称只是“问题一”“问题二”，没有可验证描述。
- `required_outputs` 为空或只写“得到结果”。
- 题面中的评价标准没有进入任何子问题。
- 为了匹配熟悉的方法而改变题目动作或输出。
