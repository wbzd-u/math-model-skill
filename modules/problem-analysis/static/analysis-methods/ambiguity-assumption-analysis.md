# 第六轮：歧义与假设分层

## 目标

显式区分未知信息、解释冲突和建模假设，防止 Agent 用无声推断填补题目缺口。

## 必须执行

1. 对所有无法唯一解释的题意、字段、单位和规则建立 `amb-*` 条目。
2. 为歧义选择处理方式：询问用户、核验规则、核验数据、留给建模或保持未解决。
3. 将假设区分为题目给定、数据支持、领域惯例和建模阶段提案。
4. 标记每项假设影响的子问题及是否需要验证。
5. 判断歧义是否改变问题定义；若改变，则标记为 `blocking`。
6. 对定义性歧义记录至少两种合理解释、每种解释影响的目标/可行域/评价/数据/输出，以及裁决所需证据；将其反向关联到所有受影响子问题的 `unresolved_ids`。

## 提示问题

- 如果采用另一种合理解释，子问题、目标或约束会改变吗？
- 该条件来自题面、数据、官方规则还是 Agent 推测？
- 这个假设现在必须确定，还是应留给模型选择阶段比较？
- 用户回答、数据检查或规则核验中的哪一种能解决它？

## 输出

完成 `ambiguity-register.yaml` 和 `assumption-register.yaml`。

## 失败信号

- 使用“默认认为”但没有假设条目。
- 把建模阶段提案标记为 `confirmed`。
- 影响问题定义的歧义仍标记为非阻断。
- 同一陈述同时被登记为题面事实和提案假设。
- 会改变问题定义的歧义仅标记为 `open`，或没有竞争性解释和裁决证据。
