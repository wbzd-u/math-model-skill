# 第五轮：子问题依赖分析

## 目标

建立子问题之间明确的信息流，识别顺序、共享状态和潜在循环依赖。

## 必须执行

1. 将所有子问题 ID 放入依赖图节点。
2. 对每条依赖边说明传递的产物和依赖原因。
3. 区分数据共享、参数传递、结果扩展和评价复用。
4. 核对子问题的 `upstream_ids` 与图中的入边一致。
5. 检查循环依赖；无法消除时建立阻断歧义。
6. 在每个子问题的 `semantic_contract.couplings` 中记录题问内部的时间、空间、资源、逻辑、信息或全局—局部耦合；这不等同于题问级依赖边。

## 提示问题

- 后一问是否需要前一问的参数、预测、排序或方案？
- 哪些子问题可并行，哪些必须顺序执行？
- 后一问是新问题，还是前一问在不同情景下的扩展？
- 多个子问题是否共享同一符号、数据口径或评价标准？
- 某个输出是否在产生前就被上游使用？

## 输出

填写 `dependency-graph.yaml`，同步更新 `subproblems.yaml.upstream_ids`。

## 失败信号

- 图中边没有说明传递产物。
- `upstream_ids` 与依赖图不一致。
- 所有子问题被默认当作独立任务。
- 存在循环依赖但未登记处理方式。
