# 第七轮：对抗性完整性审查

## 目标

以严格竞赛评审和下游建模者视角攻击题目分析初稿，发现遗漏、误读、越界和不可交接内容。

## 审查维度

逐项给出 `PASS`、`WARN` 或 `FAIL`，并附证据：

1. `requirement_coverage`：所有动作词和交付要求是否进入追踪表。
2. `source_traceability`：关键事实是否有可定位来源。
3. `entity_consistency`：实体、变量、符号、单位和粒度是否一致。
4. `subproblem_completeness`：每个子问题是否有输入、输出和评价标准。
5. `data_alignment`：每个子问题是否有数据状态记录。
6. `dependency_consistency`：依赖图与子问题定义是否一致。
7. `fact_assumption_separation`：事实、歧义和假设是否分离。
8. `scope_fidelity`：是否擅自扩大、缩小或重写题目范围。
9. `method_leakage`：是否提前选择具体模型、算法或求解器。
10. `semantic_contract_completeness`：每个子问题是否形成输入/决策/事件或状态/成功判定/聚合/输出闭环，并有验收测试。
11. `structural_difficulty`：动态、耦合、信息时点、聚合或范围难点是否有证据、影响和误读后果。
12. `definition_impact_gating`：改变问题定义的歧义是否有竞争性解释、正确反向引用并阻断交接。

## 必须提出的反向问题

- 哪一句题面尚未被任何要求条目覆盖？
- 哪个结论无法指回原题、附件或官方规则？
- 哪个子问题即使完成，也仍不能回答原题要求？
- 哪项数据缺口会让下一阶段误选模型？
- 哪个假设被偷偷写成事实？
- 是否因为想到某种方法而改变了问题定义？
- 哪个成功判定、聚合口径或硬约束一旦换一种合理解释，就会让答案改变？
- 哪个子问题在变量齐全时仍因共享资源、事件顺序或信息时点而不能独立处理？

## 输出

填写 `analysis-audit.yaml`。任何 `FAIL` 都阻止 `PASS`；未解决但不改变问题定义的 `WARN` 可以产生 `PASS_WITH_OPEN_ITEMS`。存在未解决定义性歧义时，结论必须为 `BLOCKED`。

只记录可检查的发现、证据和修正动作，不输出或要求隐藏的内部思维过程。
