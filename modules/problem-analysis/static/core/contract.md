# 题目分析契约

## 目标

把题目和附件转化为可追溯、机器可读、足以支持下一阶段建模选择的问题定义。分析质量以“是否完整理解题目”为准，不以是否提出复杂模型为准。

## 强制原则

1. **来源优先**：每项题意判断记录题面、附件或官方规则来源。
2. **输入只读**：原始文件不修改，转换文件写入项目目录。
3. **事实分层**：区分题目给定、数据支持、领域惯例、待验证假设和未知信息。
4. **先拆题后分类**：先确定子问题的输入输出，再标记任务结构。
5. **不选方法**：本阶段只描述问题，不推荐具体模型、算法或求解器。
6. **显式不确定性**：无法确认的信息进入歧义登记，不静默猜测。
7. **文件交接**：下游只依赖固定交付物，不依赖聊天记忆。
8. **语义闭环**：每个实质子问题必须说明输入/决策/事件或状态/成功判定/评价聚合/输出的关系。
9. **结构难点证据化**：难点必须说明证据、受影响维度和误读后果；不能用“复杂”“非线性”等空泛标签代替。
10. **定义性歧义阻断**：未解决且会改变目标、硬约束、成功判定、聚合口径、核心实体或输出接口的歧义，必须标记为 `blocking` 并使阶段状态为 `BLOCKED`。

## 官方规则优先级

1. 用户提供的当届官方文件。
2. 目标竞赛官方渠道获得的当届文件。
3. 题目正文中的明确要求。
4. 历年规则和经验只能作为待核验提示，不能覆盖前三项。

## 来源锚点

根据输入类型记录最具体的可用锚点：

- 文档：文件名、页码、章节或段落。
- 表格：文件名、工作表、字段或单元格范围。
- 图片：文件名、图号和可定位区域。
- 文本：文件名和行号或段落。
- 用户消息：标记为 `user-provided` 并概括原始要求。

## 输出约束

- YAML 文件使用 UTF-8，并保持为 JSON-compatible YAML。
- ID 使用稳定的小写形式，例如 `req-001`、`sp-01`、`data-01`。
- 所有跨文件引用必须指向实际存在的 ID。
- 报告中的内容应由结构化文件生成或与其一致。
- 每个子问题的任务标签、变量引用、验收测试和结构难点必须可跨文件追溯。
