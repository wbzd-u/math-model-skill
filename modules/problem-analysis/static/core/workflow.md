# 题目分析工作流

## 1. 冻结输入

- 列出题目、附件、规则和用户补充说明。
- 记录文件路径、格式、大小和 SHA-256。
- 不修改原始文件。

## 2. 识别竞赛和范围

- 确认比赛、届次、题号、语言和用户要求的分析范围。
- 找到当届规则来源；找不到时在歧义登记中记录。
- 记录正式交付物和题目要求的评价目标。

## 3. 执行七轮分析协议

依次读取 `manifest.yaml.analysis_sequence` 中的文件。每轮只处理其指定任务并更新对应产物；不得跳过对抗性审查。

以下步骤概括七轮协议，具体提示问题、失败信号和输出要求以各方法文件为准。

## 4. 逐条解析题面

将题面内容拆成独立要求，按以下类别登记：

- `background`：背景说明。
- `fact`：题目明确事实。
- `input`：给定数据或参数。
- `objective`：需要回答或优化的目标。
- `constraint`：必须满足的限制。
- `output`：必须提交的结果形式。
- `evaluation`：评价结果好坏的标准。

每条要求都要有来源锚点，并关联一个或多个子问题。

## 5. 建立实体变量图

- 识别实体、属性、关系和变量角色。
- 记录单位、取值范围、时间粒度、空间粒度和来源。
- 对同名异义、异名同义和单位冲突建立歧义项。

## 6. 审计附件和数据

- 识别文件、工作表、字段、单位、粒度、时间和空间范围。
- 检查缺失、重复、异常、编码、公式、关联键和潜在数据泄漏。
- 只描述已观察到的数据特征，不在本阶段决定清洗算法。
- 对无法读取或含义不明的字段建立歧义项。

## 7. 拆分子问题

对每个子问题记录：

- 原题要求和来源。
- 输入及其来源。
- 需要产生的输出。
- 目标、约束和评价标准。
- 上游依赖和下游使用者。
- 尚未解决的问题。

子问题 ID 必须稳定。只分析部分题目时，明确 `scope` 和未覆盖范围。

同时建立 `semantic_contract`：给定/外生量、可控量、结果量、事件或状态关系、信息时点、成功判定、评价聚合、可行性摘要、边界/终止、耦合和验收测试。随后依据子问题 `task_tags` 加载匹配的结构检查卡。

## 8. 对齐数据与任务

- 为每个子问题列出完成任务所需的数据和参数。
- 标记为可用、可推导、需要外部数据、需要假设、缺失或不适用。
- 记录数据缺口是否改变问题定义或只影响后续精度。

## 9. 建立依赖关系

- 把每个子问题作为节点。
- 用有向边表示结果、参数或数据依赖。
- 记录边上传递的中间产物及原因。
- 检查循环依赖；无法消除时登记为阻塞问题。
- 除题问级依赖外，登记时间、空间、资源、信息、网络和全局—局部评价耦合。

## 10. 标记任务结构

可使用以下结构标签描述问题，但不得据此选择算法：

`prediction`、`evaluation`、`optimization`、`classification`、`simulation`、`network`、`mechanism`、`policy-analysis`、`uncertainty-analysis`、`mixed`。

## 11. 登记歧义和假设

- 歧义项说明问题、来源、影响、处理方式和状态。
- 假设项区分题目给定、数据支持、领域惯例和留给建模阶段的提案。
- 本阶段不得把提案假设写成已确认事实。
- 对会改变问题定义的歧义记录至少两种竞争性解释、各自影响和裁决所需证据；未解决时必须 `blocking`。

## 12. 对抗性审查

- 从要求覆盖、来源追踪、变量一致、子问题完整、数据对齐、依赖一致、事实假设分离、范围忠实、方法泄漏、语义契约完整性、结构难点和定义性歧义门禁十二个维度审查。
- 将每项检查写入 `analysis-audit.yaml`。
- 任何 `FAIL` 都阻止阶段通过。

## 13. 生成报告

报告至少包括：

1. 任务范围和比赛信息。
2. 输入与数据概览。
3. 逐问拆解和依赖关系。
4. 目标、约束、评价标准和交付物。
5. 歧义、风险和待验证假设。
6. 阶段门禁结论。

## 14. 校验与停止

运行严格校验。通过后输出交接包并停止，不进入模型选择。未通过时区分：

- 可由当前阶段修正的问题：立即修正。
- 需要用户、规则或附件补充的问题：保留阻塞项并说明。
- 属于下一阶段的问题：以待处理项交接，不在本阶段解决。
