# 第五轮：数学模型规格

## 目标

把选定方法链转化为 solver 无需猜测问题含义即可实现的数学定义。方法名不能代替公式、变量角色和约束。

## 规格化顺序

1. **冻结语义**：从上游复制引用而非改写成功判定、评价聚合、硬约束、输出要求和信息时点，写入 `immutable_semantics`。
2. **统一记号**：将已有 `var-*` 映射为数学符号；新增派生量使用 `dvar-*`，说明定义和单位。
3. **登记新假设**：每项 `sel-asm-*` 写明依据、影响范围、验证方法和状态。假设不能伪装成题面事实。
4. **定义组件**：每个 `cmp-*` 写明候选来源、角色、子问题、输入、输出、目的和正式数学定义。
5. **写全数学结构**：明确目标函数或评价函数、硬约束、状态/事件关系、边界/终止条件和参数来源。
6. **建立模型链**：组件作为节点，阶段产物作为边；输入输出必须相容。允许迭代、反馈或不动点环，但必须说明收敛、终止和失败条件。
7. **执行量纲和极端情形检查**：检查单位、符号域、零值、上下界、空集、不可行情形和时间顺序。

## 提示问题

- 只看规格，不看候选说明，能否写出同一数学问题？
- 每个输出变量由哪个关系产生，单位和定义是否明确？
- 每个目标、约束和参数能否指回题面、数据、证据或显式新假设？
- 哪些约束是硬约束，哪些只是偏好、正则项或评价指标？
- 多阶段组件传递的是状态、参数、预测、可行域还是评分？
- 基线、主模型和备选是否使用同一成功判定和评价口径？

## 输出

填写 `model-specification.yaml`。每个正式组件必须引用 `model-decision.yaml` 中相应角色的候选，并覆盖至少一个选定子问题。

## 禁止事项

- 不写 Python、MATLAB、R 代码或库调用。
- 不指定未经论证的参数值、随机种子或求解器技巧。
- 不写任何未运行的数值结果、精度或性能结论。
- 不用“采用某模型”代替正式定义、目标、约束和参数来源。
