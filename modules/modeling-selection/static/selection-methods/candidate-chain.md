# 第三轮：候选方法链

## 目标

生成少量结构上真正不同、能够从输入走到必交输出且可以被证伪的方法链。

## 生成协议

1. 先写每条路线必须承担的能力角色，再填写具体方法族：状态表示、参数估计、预测、评价、优化、仿真、聚合、不确定性或验证。
2. 为每条链写清：
   - 输入、阶段产物、最终输出和阶段依赖；
   - 覆盖的 `sp-*`、`req-*` 和 `sp-*:diff-*`；
   - 数据需求、可识别条件和新增假设；
   - 证据、迁移限制、验证方式、实现成本和解释性；
   - 会让整条路线失效的拒绝条件。
3. 优先形成一个低复杂度、可解释的最小可行路线；只有比较确有意义时再增加基线。
4. 候选差异必须来自机制、信息利用、目标处理、约束处理或不确定性处理。只改变超参数、求解器、精度或可视化不算新候选。
5. 候选数量服从问题需要。一个论证充分的候选可以继续；存在实质不同的机制、假设或风险处理时再增加候选，不制造同义模型。

## 逐链压力测试

- 从原始输入开始，能否逐阶段得到题目要求的输出？
- 哪个参数无法从题面、数据、文献或校准步骤获得？
- 哪项硬约束只是写在文字里，却没有进入链条？
- 数据量、维度、噪声、时间顺序和资源耦合是否满足方法前提？
- 如果关键假设不成立，链条在哪里失败，如何观测到失败？
- 该路线如何处理题目分析中最重要的结构难点？

## 输出

填写 `candidate-methods.yaml.candidates`。每条链使用稳定 `cand-*` ID，每个阶段使用 `stage-*` ID，并只引用存在的上游与证据 ID。

## 失败信号

- 从 `prediction`、`optimization` 等标签直接跳到某个算法名称。
- 候选只有方法名，没有输入到输出的能力链。
- 多个候选只是同一模型的不同参数、求解器或验证方式。
- 主张方法“更准确”或“更优”，但尚未运行实验且没有适用性证据。
