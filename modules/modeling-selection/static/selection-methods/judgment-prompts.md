# 候选路线判断提示

对每条实质候选链逐项回答。答案应进入候选比较、验证计划、路线审计或人机决策台账，而不是写成泛泛的“创新点”。

1. 当前子问题的结构难点具体缺少什么能力：表示状态、利用信息、满足约束、量化不确定性、形成决策还是验证结论？
2. 更简单、透明的可行路线能否完成题目要求？如果不能，缺少什么能力，新增复杂度如何补足，哪个 `val-*` 能真正判断其收益？
3. 每个模型组件是否由一个可定位的题目结构、数据条件或失败模式动机驱动？若移除该组件，哪一项需求、约束或验证会受影响？
4. 当前数据、时间预算和实现环境是否支持该路线的可识别性、计算规模和验证？哪些参数、先验或外部资料仍需要人工选择或核验？
5. 这条路线的所谓创新是解决了当前题目的非显然瓶颈，还是仅更换了算法名称、增加了超参数或叠加了常见模块？收益是否可能小于新增限制？
6. 竞赛规则、用户目标或现实约束是否改变收益、风险、解释性和交付速度之间的取舍？若不存在技术唯一解，创建人机决策条目而不是由 Agent 假定偏好。

不要求每条路线都有基线、备选、外部文献或创新性主张。只保留能够改变当前选择或验证设计的回答。
