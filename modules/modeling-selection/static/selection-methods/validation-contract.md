# 第六轮：验证与实施契约

## 目标

在看到数值结果之前定义“如何知道模型做对了”，并把数学规格转换为 solver 可执行但不可静默改写的合同。

## 验证设计

先建立“结论/风险 → 测试”映射。至少覆盖：

- 每个主模型组件；基线组件只在需要比较时覆盖；
- 每个选定要求和结构难点；
- 硬约束、边界、不可行情形和事件顺序；
- 关键参数与新增假设；
- 需要声称优于基线的主要结果。

按需要选择基线比较、约束检查、误差、敏感性、鲁棒性、不确定性、边界、对抗性和复现测试。每个 `val-*` 必须在运行前写明目标主张、步骤、指标、通过标准和失败动作；不得写“结果合理”或事后调整阈值。

## 实施契约

1. 只批准 `model-specification.yaml` 中存在的组件。
2. 列出范围、输入字段、参数来源与取值政策、执行阶段、阶段输入输出和必需产物。`input_refs` 优先使用 `data-*`、`var-*`、`param-*` 或可追溯的前序阶段产物 ID；不能只写含义不明的自然语言名称。
3. 将题目定义、目标、硬约束、核心假设、评价口径和输出要求列入 `immutable_items`。
4. 将语言、库、数据结构、数值算法、容差和性能优化等允许选择的事项列入 `solver_discretion`；这些选择不得改变不可修改项。
5. 为不可实现、数据不足、参数不可识别、约束冲突和验证失败定义反馈触发器及证据要求；默认反馈路径为 `solver/modeling-feedback.yaml`。
6. 在 `model-specification.yaml` 和 `validation-plan.yaml` 最终写定后计算 SHA-256，并连同 `intake-check.yaml` 哈希写入 `input_snapshot`。

## 提示问题

- 每项主要结论失败时，哪个测试会发现？
- 通过标准是否客观、可计算，并在看到结果前确定？
- solver 能否仅凭契约找到所有输入、参数、阶段和输出？
- 哪些实现自由不会改变问题，哪些改变必须退回本模块审批？
- 基线与主模型是否在同一数据、指标和约束下比较？

## 输出

依次完成 `validation-plan.yaml`、`implementation-contract.yaml`、`modeling-selection-audit.yaml` 和最终报告。探索中运行 `--mode explore`；准备求解时运行 `--mode handoff`。本模块不自行执行代码，诊断性试算由 solver 返回证据后再迭代模型。

## 失败信号

- 验证只覆盖平均误差，没有约束、边界或失败情形。
- 通过标准依赖尚未生成的结果或使用“较好”“合理”等模糊词。
- 契约加入规格中不存在的目标、变量、约束或参数。
- solver 被允许自行改变模型含义，或不可实现时可以静默换模型。
- 契约快照哈希不是基于最终版本计算。
