# 第一轮：准入与范围

## 目标

把题目分析交接包压缩为本阶段不可偏离的选择范围。此轮只确认“要解决什么”和“什么不能改变”，不提出算法。

## 必须执行

1. 读取 `intake-check.yaml`，确认上游版本、门禁状态、文件哈希、允许范围和用户请求范围。
2. 对每个选定子问题提取：
   - `source_requirement_ids` 与必交输出；
   - `success_predicate`、验收测试和评价聚合；
   - 外生量、可控量、结果量、硬约束和信息时点；
   - 可用数据、可推导数据、缺失数据和字段问题；
   - `difficulty_drivers`、开放歧义和上游依赖。
3. 检查每个选定子问题是否都在 `allowed_subproblem_ids` 中。
4. 将题目事实、上游假设和本阶段拟议假设分开；本轮不得新增假设。
5. 分别记录可探索范围和可正式 handoff 范围，再进入证据检索。

## 提示问题

- solver 最终必须产出什么，如何判断它确实回答了原题？
- 哪些条件改变后，求解的就不再是同一个问题？
- 哪些结构难点必须由主模型处理，哪些可以由验证步骤处理？
- 哪个数据缺口只影响精度，哪个会让方法不可识别或问题不可定义？
- 子问题能独立选择方法，还是必须与上游或共享资源联合设计？

## 输出

确认并更新 `modeling/intake-check.yaml` 的选择范围；在 `modeling-selection-report.md` 中形成逐子问题范围摘要。不要复制或改写 `analysis/` 文件。

## 失败信号

- 完全缺少要求或子问题时仍声称已经形成建模路线。
- 将探索范围误写成可正式 handoff 范围。
- 只复述题号，没有写清成功判定、数据状态或结构难点。
- 通过常识、历史经验或临时假设掩盖定义性歧义，而没有建立竞争性解释和验证条件。
