# 人机决策台账协议

## 目的

台账只记录无法由题面、数据、模型契约或验证结果唯一决定的**高影响选择**。它让人保留研究判断权，同时使选择、理由、待验证条件和回退路径可追溯。

不要为低风险实现细节创建条目，也不要把每个模型参数都交给人工选择。

## 何时创建条目

创建条目的典型情形：

- 问题范围、实体解释或不可接受假设存在多个合理版本；
- 目标之间必须取舍，例如收益与风险、准确性与可解释性、短期与长期；
- 是否使用外部数据、领域先验或高成本计算会实质改变结论；
- 多条主路线都科学可行，但选择取决于现实偏好或项目资源；
- 最终结论的适用范围、建议力度或创新性表述需由人承担责任。

不创建条目的典型情形：文件解析、字段审计、常规库选择、等价数值实现、图表字号、日志路径、已被官方规则唯一确定的格式。

## 工作方式

1. 从模板建立 `PROJECT_ROOT/decisions/human-decision-ledger.yaml`；只在当前项目存在高影响选择时创建。
2. 为每项选择写清问题、选项、证据、风险、推荐、验证条件和失败回退。证据引用 `req-*`、`sp-*`、`data-*`、`cand-*`、`val-*`、`run-*`、`claim-*` 或可定位的官方来源。
3. 状态设为 `awaiting-human` 后，向用户简洁呈现选项与推荐。不要代替用户选择。
4. 收到明确选择后，记录选择、理由和时间，将状态改为 `selected`。人可以选择一个选项、要求并行试验，或否决全部选项。
5. 只有所有依赖某项选择的条目均为 `selected` 或 `not-needed`，相关路线才可进入正式 handoff、`solve` 或 `final`。

## 状态与边界

- `awaiting-human`：允许继续不依赖该项的审计、探索和 probe 准备；阻断依赖它的正式提交。
- `selected`：下游必须遵守选择及其验证条件；若失败回退触发，建立新条目或重新打开该条目。
- `not-needed`：记录为什么题意、数据或证据已唯一决定此事。
- `superseded`：保留旧条目及原因，不删除历史决定。

Agent 推荐必须标为推荐，不得伪装为人工意见。用户未回复、聊天历史中的模糊倾向或 Agent 默认策略都不是 `selected`。
